**フェーズ4-Bのスコープ**: バイラルループとサマリー自動化。

## セキュリティ運用ルール(絶対遵守)

`docs/knowledge/03a_Webアプリ_メイン_認証セキュリティ.md` の全ルールを厳守。

### このフェーズでの特に重要なポイント
- シェアカード生成APIは**認証必須**にする(他人のデータでカードを作れないように)
- 生成画像にはユーザー固有のIDなどPIIを含めない(街並み・累計チェック数・建物数・住人数のみ)
- 画像生成のクエリパラメータに `user_id` を直接使わない(URLに露出する)
  - 代わりに、サーバー側でCookieセッションから`auth.uid()`を取得する設計に
- 月次自動生成は他人の月次データを取れないよう、必ず`auth.uid() = user_id`のRLSポリシー越しに動かす

### Security Audit First
コードを提案する前に、その実装が「他のユーザーの画像を生成できないか」「URLを推測されて他人のシェアカードが取れないか」を内部で監査し、結果を短く報告してください。

## 技術選定の確定事項

| 項目 | 採用 |
|---|---|
| 画像生成 | Next.js `ImageResponse`(内部はSatori) |
| 描画方式 | TSXコンポーネント(Flexboxのみ) |
| デプロイ互換性 | Cloudflare Workers + OpenNext で動作 |
| サイズ | 16:9 横長(X用) と 1:1 正方形(Instagram/note用)の両方を自動生成 |

### Satoriの制約(重要、Claude Codeは認識して実装する)
- **Flexbox のみサポート**(Gridや絶対配置の一部は不可)
- **Framer Motion は動かない**(静止画なので不要)
- **一部のCSSプロパティ非対応**(`transform: scale` など)
- **フォントは明示的にロード**(`@vercel/og` がやってくれる)
- **画像は base64 埋め込み or 静的アセット**

## 仕様詳細

### 1. シェアカード生成基盤

#### エンドポイント
- `app/api/share-card/route.ts`(または `app/share-card/[size]/route.ts`)
- HTTPメソッド: GET
- 認証: Cookie ベースのSupabaseセッションから`auth.uid()`を取得
- クエリパラメータ: `size`(`landscape` or `square`)、`period`(任意、例: `2026-04` で「Month 4」表示)

#### レスポンス
- `Content-Type: image/png`
- Next.js `ImageResponse` でPNG画像を返す

#### 街並みデータの取得
- サーバー側で認証ユーザーのデータを取得:
  - `profiles.started_at`(Day計算)
  - `milestone_progress.current_era`(現在の時代)
  - 累計チェック数(`daily_checks` の集計)
  - 建物数・住人数(累計チェック数から計算)
  - 現在の季節(`user_settings.current_season`)

#### キャッシュ
- 画像生成は重いので、Cache-Controlヘッダーで適切にキャッシュ
- ただしユーザーデータが変わったら更新される必要があるので、`Cache-Control: private, max-age=300`(5分)程度

---

### 2. 街並みのSatori互換版コンポーネント

フェーズ3で作った街並みコンポーネントは Framer Motion や `position: absolute` を使っている可能性があります。
**Satoriで動く別バージョンを作る必要があります**。

#### 設計方針
- 既存の `components/city/CityScene.tsx` には触らない
- 新しく `components/share/StaticCityScene.tsx` を作る
- 同じ `era`/`buildingCount`/`villagerCount`/`season`/`seed` を受け取って、Satori互換で描画

#### Flexbox による街並みの組み方
- 横並び`flex flex-row` で建物を並べる
- 各建物は固定サイズの`div`(色 + 形をCSSで)
- 住人は建物の隙間に小さなdivで配置
- 季節エフェクトは半透明の色付きオーバーレイで表現(回転する太陽など複雑なものはシンプル化)

#### 元コンポーネントとの一貫性
- `lib/city/assets.ts` の定数(色・サイズなど)を共有して使う
- 建物の見た目はダッシュボードと同じ印象になるように

### 3. シェアカードのレイアウト

#### 16:9 横長(1200 x 675px推奨)

```
┌─────────────────────────────────────────────┐
│                                              │
│   Month X                          Day NNN   │
│                                              │
│  ┌──────────────────────────────────────┐  │
│  │                                      │  │
│  │         街並み(StaticCityScene)     │  │
│  │                                      │  │
│  └──────────────────────────────────────┘  │
│                                              │
│  累計チェック: NN個   建物: N棟   住人: N人   │
│                                              │
│                       cityup.example.com →   │
└─────────────────────────────────────────────┘
```

#### 1:1 正方形(1080 x 1080px推奨)

```
┌───────────────────────────┐
│         Month X            │
│         Day NNN            │
│                            │
│ ┌────────────────────┐    │
│ │                    │    │
│ │     街並み          │    │
│ │                    │    │
│ └────────────────────┘    │
│                            │
│   累計チェック: NN個        │
│   建物: N棟  住人: N人      │
│                            │
│      cityup.example.com    │
└───────────────────────────┘
```

#### 表示の温度感
- 「Month X」「Day NNN」のように事実を淡々と表示
- 「すごい!」「達成!」などの言葉は使わない
- 数字を並べるだけで、見た人が「これ何のアプリ?」と気になる程度の控えめさ
- アプリURLは小さく、ただし読める大きさで右下に

### 4. シェアカード生成ボタン

#### ダッシュボードに設置
- ヘッダー付近に「シェアカードを作る」ボタン
- アイコン + 短いラベル(例: 📤 シェア)

#### 振り返り画面に設置
- 振り返り画面の上部に同じボタンを配置

#### クリック時の挙動
1. 「シェアカードを作っています」のローディング表示(モーダル)
2. サーバーから16:9と1:1の両方のPNGをfetch
3. プレビューモーダル表示(後述)

---

### 5. シェアカードプレビューモーダル

#### 表示内容
- 16:9 と 1:1 のプレビューを並べる(タブ or 切り替えボタン)
- 大きく見える形でプレビュー
- 下部にアクションボタン:
  - **ダウンロード**(現在表示しているサイズのPNG)
  - **Xで共有**(`https://twitter.com/intent/tweet?text=...&url=...` で開く、画像は別途ユーザーが添付)
  - **閉じる**

#### Xで共有の挙動
- Web Share APIが使えるなら優先(`navigator.share`)
- 使えない場合は `https://twitter.com/intent/tweet` の URL を開く
- text には「Day NNN、街がここまで育ちました。」程度の控えめな定型文(編集可能であることをユーザーに伝える)
- url にはアプリのURL

#### ダウンロードの挙動
- PNGとして保存(`<a download>` で実現)
- ファイル名: `cityup_dayNNN_landscape.png` のような形式

---

### 6. マイルストーン到達時の提案トースト

#### 発火タイミング
- Day 1/7/30/90/180/270/365 のマイルストーン演出が**完了してから2秒後**に表示
- 一度のセッションで同じマイルストーンには再表示しない

#### トーストの内容
- メンターのアイコン + 短いメッセージ
- 例: 「シェアカードを作りますか?」
- ボタン:
  - 「作る」 → シェアカードプレビューモーダルを開く
  - 「あとで」 → トーストを閉じる(以降このセッションでは出ない)

#### 表示時間
- **10秒間表示**
- ユーザーが何もしなければ10秒後にフェードアウト
- ユーザーが「あとで」を押すと即座に閉じる

#### 押し付けない設計の確認
- 「作る」がデフォルトでハイライトされないこと(等価扱い)
- 強い色や点滅は避ける
- 一度閉じたら次のマイルストーンまで再表示しない

---

### 7. 月次成果入力後の自動生成(月次サマリー機能)

#### 発火タイミング
- ユーザーが**月次成果入力フォームを送信完了した直後**
- 入力完了画面の後に、自動で先月分のシェアカードを生成して表示

#### 解放条件
- **Day 90 以降**(Day 90で月次サマリー自動生成が解放される設定、過去フェーズで合意済み)
- Day 90未到達のユーザーは、通常通り入力完了で終わる(自動生成しない)

#### フロー
1. 月次入力完了
2. (Day 90 以降なら)「先月のサマリーを作成中...」のローディング表示
3. シェアカードプレビューモーダルが自動表示
4. ユーザーは保存/共有/閉じるを選べる

#### `period` パラメータの活用
- 自動生成では `period=YYYY-MM` を指定して、その月の街並み状態(月末時点)で生成
- 月末時点のスナップショットを使う(過去比較機能と同じスナップショット計算を流用)

## DB変更

このフェーズで新規テーブルは基本不要。
ただし以下のような小テーブルを追加することも検討:

#### 候補: `share_card_history`(シェアカード生成履歴)
```sql
CREATE TABLE share_card_history (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid REFERENCES auth.users NOT NULL,
  trigger text NOT NULL,  -- 'manual', 'milestone', 'monthly_auto'
  size text NOT NULL,     -- 'landscape', 'square'
  period text,            -- 'YYYY-MM' (任意)
  created_at timestamptz DEFAULT now()
);
```

**追加するかはあなたが判断**してください。バイラルループの効果測定に使えますが、必須ではありません。
