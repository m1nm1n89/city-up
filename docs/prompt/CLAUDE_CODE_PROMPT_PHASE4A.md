# Claude Code 用プロンプト ─ フェーズ4-A: 振り返り画面 + 過去比較機能

このプロンプトは VSCode 内の Claude Code 拡張機能に渡す指示書です。
フェーズ3までで街並みとマイルストーン演出が完成しています。
フェーズ4-Aでは**振り返り画面**と**過去比較機能**を実装します。

---

## あなたの役割

経験豊富な Next.js + React + Framer Motion 開発者として、ユーザーが「過去の自分」と対話できる画面を作ってください。
セキュリティ運用ルールはフェーズ1〜3と同様に厳守してください。

開発を始める前に、必ず `docs/` 配下のMDファイルを全て読み、フェーズ1〜3で作られたコードベースの構造も把握してから着手してください。

```
docs/
├── 03a_Webアプリ_メイン_概要.md          (コンセプト・体験設計)
├── 03a_Webアプリ_メイン_仕様詳細.md      (マイルストーン・コイン・入力・UI)
└── 03a_Webアプリ_メイン_認証セキュリティ.md (認証方式・Supabaseセキュリティ・技術スタック)
```

特にフェーズ3末に整備した `lib/city/` 配下の構造(eraを引数で切り替えられる関数)を活用してください。

---

## このフェーズで作るもの

**フェーズ4-Aのスコープ**: 振り返り体験の中核。

1. **グローバルナビゲーション**(ホーム / 振り返り / 設定 の3項目)
2. **振り返り画面の基本UI**
3. **時代スライダー**(原始〜未来、未到達時代もプレビュー可能)
4. **進捗スライダー**(Day 1〜今日までの過去の自分の街を見られる)
5. **季節ボタン**(振り返り画面でも切り替え可能)
6. **過去比較機能**(Day 30で解放、数字での比較表示)

**やらないこと**: シェアカード生成・月次サマリー・マイルストーン提案UI(これらはフェーズ4-B)

---

## セキュリティ運用ルール(絶対遵守)

`docs/03a_Webアプリ_メイン_認証セキュリティ.md` の全ルールを必ず守ってください。

### このフェーズでの特に重要なポイント
- 過去のチェック履歴・週次目標・月次成果を取得するクエリは、必ず`auth.uid() = user_id`のRLSポリシーで他人のデータが見えないようにする
- 振り返り画面が読み取るデータも、Server Componentで認証済みSupabaseクライアントを使う
- 過去比較で使う数値も**サーバー側で集計**(クライアントで `total_count` などを偽装できる経路を作らない)

### Security Audit First
コードを提案する前に、その実装が「他のユーザーのデータが漏洩しないか」を内部で監査し、結果を短く報告してください。

## 仕様詳細

### 1. グローバルナビゲーション

#### 配置
- ヘッダーに3つのリンクを設置
- スマホ表示でも崩れない構造(必要ならボトムナビ風 or ハンバーガー)

#### 項目
| 項目 | 遷移先 | 概要 |
|---|---|---|
| ホーム | `/dashboard` | 通常画面(現在の街並み、チェック入力など) |
| 振り返り | `/reflection` | 振り返り画面(このフェーズで新規実装) |
| 設定 | `/settings` | 既存(チェック候補編集など) |

現在表示中のページがハイライトされるようにする。

---

### 2. 振り返り画面の基本UI

#### URL
- `/reflection`(認証必須)

#### 画面構成
- 上部: ヘッダー(グローバルナビと一体)
- メイン: **街並み表示エリア**(ダッシュボードと同じ大きさ)
- 街並みの下: **コントロールパネル**
  - 時代スライダー
  - 進捗スライダー
  - 季節ボタン
- さらに下: **過去比較セクション**(Day 30未到達ならロック表示)

#### 街並み描画
- ダッシュボードと同じ`CityScene.tsx`を再利用
- ただし**props で era / day / season を受け取れる**ように汎用化する
- 現在の状態ではなく、コントロールパネルで指定された状態を表示

---

### 3. 時代スライダー

#### 仕様
- 6段階のスライダー: 原始 / 古代 / 中世 / 近代 / 現代 / 未来
- ユーザーが**未到達の時代もプレビュー**可能(モチベーションを上げるため)

#### UI
- 横スライダー(つまみをドラッグで切り替え)
- 各時代の名前と「到達日」「未到達」を表示
- 例:
  ```
  ● 原始 (Day 1〜)
  ● 古代 (Day 30 達成済)
  ○ 中世 (Day 90 未到達)
  ○ 近代 (Day 180 未到達)
  ```
- 未到達時代を選んだ場合、街並みの上に薄く「未到達」の透かしを表示
  - 例: `<div class="opacity-70">` + 上部に「これは Day 90 で見られる景色です」のテキスト

#### データ取り扱い
- 時代スライダーで時代を変えても、それは**ユーザーの`current_era`を変更しない**
- あくまで「振り返り画面のローカルな状態」(URLクエリ or React state)
- 振り返り画面を離れたら状態は破棄してOK

---

### 4. 進捗スライダー

#### 仕様
- Day 1〜現在の累計日数までの範囲で、任意の日付の街を表示できる
- 「過去のあの日、自分の街はこうだった」を見られる

#### UI
- 横スライダー(つまみをドラッグ、または数値入力で日数指定)
- 表示: 「Day {n} / 累計 {累計日数}」
- 0%(Day 1) → 100%(累計日数)で操作可能
- 「今日」ボタンで一発で現在に戻れる

#### 計算ロジック
- 選択された Day n のスナップショットは、以下から再構築:
  - その時点での `current_era`(累計日数から逆算)
  - その時点までの累計チェック数(`daily_checks` の集計)
  - その時点での建物数・住人数(累計チェック数から計算)
- フェーズ3で「街並みはシード値で再現可能」な設計にしているので、`(user_id, accumulated_checks)` から同じ配置が描ける

#### 注意点
- 過去の街を見ているとき、画面上に「Day {n} の街並みを表示中」と明示する
- アニメーションは抑える(チェック演出やマイルストーン演出は出さない)
- 季節は別ボタンで操作

---

### 5. 季節ボタン

#### 仕様
- 春/夏/秋/冬の4ボタン
- 振り返り画面でも自由に切り替え可能
- **振り返り画面の季節選択は永続化しない**(ローカル状態のみ)
- ユーザーの`user_settings.current_season`は変更しない

---

### 6. 過去比較機能(Day 30 で解放)

#### 解放条件
- ユーザーが Day 30 に到達している(`milestone_progress.current_era !== 'primitive'`、または累計日数 >= 30)
- 未達成の場合は「Day 30 に到達すると解放されます」と表示してロック

#### 表示内容: 数値の比較

振り返り画面の下部に専用セクションを設置。
以下のような比較を**カード形式**で表示:

| 比較項目 | 表示 |
|---|---|
| 累計チェック数 | 30日前: 25個 → 今: 87個 (+62個) |
| 累計建物数 | 30日前: 1棟 → 今: 4棟 (+3棟) |
| 累計住人数 | 30日前: 0人 → 今: 2人 (+2人) |
| 月次収益(自己申告) | 30日前(該当月): 0円 → 今月: 1,200円 |
| フォロワー数(任意項目) | 30日前: 12人 → 今: 47人 (+35人) |

#### 比較期間の指定
- デフォルト: 「30日前」と「今」を比較
- ドロップダウンで他の期間も選べる:
  - 30日前 vs 今(デフォルト)
  - 90日前 vs 今(Day 90到達後に有効化)
  - 180日前 vs 今(Day 180到達後)
  - 365日前 vs 今(Day 365到達後)

#### 計算ロジック
- 「30日前」= 今日 - 30日 のスナップショット
- その時点までの累計値を `daily_checks` などから集計
- ユーザーがDay 30ちょうどの時は、「Day 1 vs Day 30」を表示

#### サーバーサイド集計
- 数値は **Server Action や RPC で取得**
- クライアント側で `daily_checks` を全件取得して数えるのではなく、Postgres側で集計
- パフォーマンス対策: 必要なら集計用のビュー or マテリアライズドビュー(MVP では集計関数で十分)

#### 表示の温度感
- 「すごい!」「成長しました!」などは使わない
- 数字を**淡々と並べる**だけでOK
- 比較自体が体験の核(数字が動かない時期の人にも「日数とチェック数だけは確実に進んでいる」と気づかせる)

## DB変更

このフェーズで新規テーブルは不要。既存テーブルへの集計クエリで対応できます。

ただし、過去のスナップショット再現のために、もし必要なら以下のような RPC 関数を作るのもアリ:

```sql
CREATE OR REPLACE FUNCTION get_user_snapshot(
  target_user_id uuid,
  target_day_offset int  -- 何日前のスナップショットか
)
RETURNS TABLE (
  accumulated_checks int,
  building_count int,
  villager_count int,
  era text,
  monthly_revenue int,
  monthly_followers int
)
SECURITY DEFINER
AS $$
  -- 集計ロジック
$$ LANGUAGE plpgsql;
```

必要かどうかはあなたが判断してください。
シンプルなクエリで済むなら無理に作らなくてOK。
