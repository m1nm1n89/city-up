## このフェーズで作るもの

**フェーズ2のスコープ**: 入力ロジックとコイン経済の完成。

1. **チェックボックス入力**(毎日、3個固定、10候補から選択)
2. **目標入力フロー**(初回の最終目標 + 毎週の小さな目標)
3. **週末の進捗記入**(達成判定 → コイン獲得)
4. **月次成果入力**(必須2項目+任意項目、催促あり)
5. **コインロジック**(獲得・上限・残高管理)
6. **dashboardの拡張**(現在のコイン・チェック状況の表示)

**やらないこと**: 街並みUI・アニメーション・マイルストーン演出・シェアカード(これらはフェーズ3以降)

## セキュリティ運用ルール(絶対遵守)

`docs/03a_Webアプリ_メイン_認証セキュリティ.md` の全ルールを必ず守ってください。

### このフェーズでの特に重要なポイント
- **新規テーブル**を追加する場合、必ず `ALTER TABLE ... ENABLE ROW LEVEL SECURITY;` (Default Deny)
- RLSポリシーは `auth.uid() = user_id` で「自分のデータのみ」
- **コイン残高の改ざんは絶対に許してはいけない**。クライアントから直接`current_coins`を書き換えできない設計にすること
- コイン更新は**サーバーサイドで計算**し、Supabaseのトリガーまたはサーバーアクション内のみで実行

### Security Audit First
コードを提案する前に、その実装が「RLSを突破されないか」「他のユーザーのデータが漏洩しないか」「クライアントから不正にコインを増やせないか」を内部で監査し、結果を短く報告してください。

## 仕様詳細

### 1. チェックボックス入力(毎日)

#### デフォルト候補10個
ユーザー初回登録時、または「候補を変更」時にプリセットされる:

1. Xに投稿した
2. noteを執筆した
3. ブログ記事を書いた
4. YouTube動画を撮影/編集した
5. Webアプリのコードを書いた
6. 競合・参考記事をリサーチした
7. 副業に関する本/教材を学んだ
8. 数字(KPI)を確認した
9. アフィリ提携/収益化の作業をした
10. アイデア出し・企画を考えた

ユーザーは10候補から3個を選択して使用する。10候補自体もカスタマイズ可能(候補編集UIを用意)。

#### コイン獲得ロジック
- チェック1個ON → +1コイン
- 3個全部ONになった瞬間 → +1ボーナス(合計4コイン)
- **1日の上限: 4コイン**(これ以上は加算されない)
- **チェックを外した場合**、その日のコインは再計算される(取り消し可)

#### 上限リセット
- **日付が変わったタイミング(0時)でリセット**(タイムゾーンはAsia/Tokyo)
- 0時を跨いだチェックは新しい日のチェックとして扱う

#### データ保存
- `daily_checks` テーブル(フェーズ1で作成済み)に保存
- 1日に1レコード(UNIQUE制約あり)
- `checks` jsonbには `{check_id: true/false}` 形式で保存

---

### 2. 目標入力フロー

#### 初回の最終目標(1回だけ)
- サインアップ後、最初に必ず入力させる(ダッシュボードに進めない)
- テンプレ or 自由記述の3択フォーマット:
  - **テンプレA**: 月XX円の収益を出す(金額を入力)
  - **テンプレB**: フォロワーXX人達成する(人数を入力)
  - **自由記述**: 自由テキスト
- `user_settings.final_goal` に保存
- 一度設定したら基本変更しない(後で「目標を再設定」UI で変更可能、ただし目立たせない)

#### 毎週の小さな目標
- **週の区切り: 日曜開始 - 土曜終了**(タイムゾーンAsia/Tokyo基準)
- 日曜の朝〜月曜午前中に「今週の目標を決めよう」と促す
- テンプレ or 自由記述の3択:
  - **テンプレA**: noteをX回投稿する(回数を入力)
  - **テンプレB**: Xに毎日ポストする
  - **自由記述**: 自由テキスト
- `weekly_goals` テーブルに保存
- `week_start_date` は日曜の日付

#### 週末の進捗記入
- 土曜の夜 or 日曜の朝に「先週の進捗を記入しよう」と促す
- 自由記述欄に「どこまでできたか」を書く
- 達成判定は**ユーザー自身がチェックボックスで判定**(「達成した」「達成できなかった」)
  - 達成と申告 → 3コイン獲得 + `achieved = true`
  - 未達成 → 何も起きない(罰なし) + `achieved = false`
- 進捗を記入しないと前週分は「未記入」のまま残る

---

### 3. 月次成果入力

#### 必須2項目
- 今月の投稿数(数値、0以上)
- 今月の収益(円、0円可)

#### 任意項目
- フォロワー数
- PV数
- その他(自由記述、テキスト)

#### 催促タイミング
- **月末(28〜31日のいずれか)** にダッシュボード上部に催促バナーを出す
- 28日からカウントし、月末日まで毎日表示
- 未入力のまま月をまたぐと、翌月の最初に「先月分を入力してください」を表示
- 入力するまでダッシュボードの他の機能(振り返り画面など)に進めないようにする(モーダルで覆う)

#### 「先月と同じ」ボタン
- 月次入力フォームに「先月と同じ」ボタンを設置
- クリックすると **全項目を一括で前月値プリセット**(ユーザーは編集可能)
- 前月のデータがない場合はボタンを無効化

#### 報酬
- 月次入力を完了 → +5コイン

#### データ保存
- `monthly_stats` テーブル(フェーズ1作成済み)に保存
- UNIQUE(user_id, year_month) で重複防止

---

### 4. コインロジック

#### 残高管理
- `coin_balance` テーブル(フェーズ1作成済み)で管理
- `current_coins`(現残高)、`total_earned`(累計獲得)、`total_spent`(累計消費)

#### 全獲得トリガー一覧

| アクション | コイン |
|---|---|
| チェック1個ON | +1 |
| 3個全達成ボーナス | +1(計4コイン/日が上限) |
| 週次目標達成と申告 | +3 |
| 月次成果入力完了 | +5 |

#### 1日の上限
- チェックから獲得できるのは **4コイン/日まで**
- 週次・月次は別カウント(上限なし)

#### 整合性の保ち方
- コイン更新は**サーバーアクション**または **Supabase Edge Function** 内で実行
- クライアントから直接 `coin_balance` を書き換える操作は**RLSで禁止**
- RLSポリシーで `coin_balance` の `UPDATE` を `auth.uid() = user_id` かつ「サービス経由のみ」にする
- 推奨実装: `RPC関数(SECURITY DEFINER)` を作り、その中でコイン更新ロジックを書く

#### 改ざん耐性のチェック
実装時、以下のシナリオで攻撃が成立しないことを確認してください:
- ブラウザの開発者ツールでSupabase APIを直接叩いてコインを増やせるか?
- チェックを連打して上限を超えるコインを得られるか?
- 同じ週次目標を複数回「達成」と申告してコインを稼げるか?

すべて防げる設計にしてください。

---

### 5. 通知/催促の設計

このフェーズではメール通知やプッシュ通知は実装しません(個人情報を持たない方針)。
**アプリ内バナー/モーダル**のみで催促します。

#### バナー表示の優先度
1. 最終目標未設定 → 強制モーダル(他に進めない)
2. 月次成果未入力(月初) → 強制モーダル
3. 月次成果未入力(月末) → 上部バナー
4. 週次目標未設定(週初) → 上部バナー(やわらかめ)
5. 週次進捗未記入(週末) → 上部バナー(やわらかめ)

「催促はするが罰しない」原則。
表示は「優しめのトーン」で。例:「今月の成果を入力してみませんか?」のようなニュアンス。

---

### 6. ダッシュボードの拡張

フェーズ1で作った仮ダッシュボードを以下に拡張:

#### 表示要素(順序)
1. ヘッダー: ユーザーネーム / ログアウトボタン
2. 催促バナー(該当時)
3. **コイン残高**(現在のコイン数を大きく表示)
4. **今日のチェックボックス**(3個、タップでON/OFF切り替え、コイン獲得アニメは仮実装でOK)
5. **今週の目標**(設定済みなら表示、未設定なら「設定する」ボタン)
6. **今月の数値**(月次入力済みなら表示、未入力なら催促)
7. 累計日数 / 連続日数(両方表示、累計大きめ・連続小さめ)

**注意**: 街並みUIはフェーズ3で実装します。このフェーズでは**テキスト/カードベース**のシンプルな画面でOK。デザインの作り込みは最小限。

## 新規追加が必要なDB構造

フェーズ1で作ったテーブルで基本足りますが、追加で必要なら以下を提案:

#### 候補: `coin_transactions`(コイン変動の履歴)
不正検知や将来のサマリー機能のために、コインの変動を全記録する案。

```sql
CREATE TABLE coin_transactions (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid REFERENCES auth.users NOT NULL,
  amount int NOT NULL,            -- 正: 獲得 / 負: 消費
  reason text NOT NULL,           -- 'daily_check' / 'weekly_goal' / 'monthly_stats' など
  reference_id uuid,              -- 関連レコード(daily_checks.idなど)
  created_at timestamptz DEFAULT now()
);
```

**このテーブルを追加するかは、あなたが必要性を判断**して提案してください。デバッグやサマリー画面で便利な反面、容量は増えます。

## 完了の定義

フェーズ2が完了したと判断する条件:

- [ ] サインアップ直後に最終目標の入力画面に遷移する
- [ ] 10候補から3個を選び、ダッシュボードでチェックON/OFFができる
- [ ] チェックでコインが増え、1日4コインで頭打ちになる
- [ ] 週次目標を設定でき、週末に進捗を書ける。達成申告で3コイン入る
- [ ] 月次成果入力で5コイン入る。月末/月初に催促バナーが出る
- [ ] 「先月と同じ」ボタンが動く
- [ ] クライアントから直接Supabase APIを叩いてもコインを不正に増やせない(改ざん耐性確認済み)
- [ ] 全テーブルでRLSが有効、`auth.uid() = user_id` ポリシーが正しく機能

完了したら「フェーズ2完了報告」を出してください。改ざん耐性テストの結果も併せて報告してください。
