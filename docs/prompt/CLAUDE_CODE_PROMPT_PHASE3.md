`docs/03a_Webアプリ_メイン_認証セキュリティ.md` の全ルールを必ず守ってください。

### このフェーズでの特に重要なポイント
- 街並みUIは**表示のみ**でDBにあるユーザーデータをそのまま読み取って描画する
- **マイルストーン到達日数の改ざんは絶対許さない**:
  - クライアントで「自分は今Day 100」と偽装させないこと
  - 経過日数は `profiles.started_at` から**サーバー側で計算**し、それをUIに渡す形にする
- デバッグツールは**RLSをバイパスできない**仕様にする(本番では完全に動かないように)

### Security Audit First
コードを提案する前に、その実装が「他のユーザーのデータが漏洩しないか」「マイルストーン進行を不正に進められないか」を内部で監査し、結果を短く報告してください。

## 描画方法と技術選定

### ライブラリ
- **Framer Motion**(アニメーション全般)
- **HTML/CSS + Tailwind**(街並みUIの構成要素、プレースホルダー)
- SVG/Canvas は今回使わない(フェーズ3はプレースホルダー方針のため)

### 街並みのプレースホルダー方針
イラスト素材はまだ用意しません。**色付きの矩形・丸・三角などCSSで描けるシンプルな図形**で街並みを構成してください。
理由: 機能・アニメーション・体験設計を確認することが優先で、絵は後から差し替えるため。

#### プレースホルダー設計の原則
- 各時代の建物は**異なる色・形・サイズ**で区別できるようにする
- 例: 古代=茶色の三角屋根、中世=赤レンガ風の長方形、未来=金属色のガラスビル風
- 後で本物の絵に差し替えやすいよう、**コンポーネント単位で抽象化**(`<Building era="ancient" type="A" />` のような形)
- イラストパスを `lib/city/assets.ts` のような定数ファイルにまとめ、差し替えやすくする

## 仕様詳細

### 1. 街並みUI

#### 画面構成
- 上部: ヘッダー(ユーザーネーム、コイン残高、累計/連続日数)
- 中央: **街並みエリア(画面の50〜60%を占める主役)**
- 下部: メンター枠(顔アイコン+セリフ)
- その下: 今日のチェックボックス3個
- 最下部: 季節切り替えボタン(春夏秋冬)

俯瞰視点で街を眺める設計。地面ライン・空・遠景の3層構造で描く。

#### 建物の配置ロジック
- 累計チェック数に応じて建物が増える
- 例: チェック数 0〜5 → 1棟、6〜15 → 2棟、16〜30 → 3棟、以降は緩やかに増加
- 建物の種類は**入力カテゴリ**(X投稿・noteなど)と連動できる設計にする
  - フェーズ3では「チェック種類ごとに違う色」程度の対応でOK
- 配置はランダムだが**再現性**を持たせる(同じユーザーIDとチェック数で同じ配置)
  - シード値ベースの疑似乱数を使う

#### 住人の配置
- 建物数に応じて住人が増える(例: 建物1棟あたり1〜2人)
- 住人もシード値ベースで配置を再現可能にする

---

### 2. 時代進化システム

#### 時代の定義(`lib/city/eras.ts` などに定数化)

| era | 表示名 | 解放日 | 視覚的特徴(プレースホルダー) |
|---|---|---|---|
| primitive | 原始 | Day 1 | 茶色の地面、緑の草、シンプルなテント |
| ancient | 古代 | Day 30 | 弥生・竪穴式風(茶色三角) |
| medieval | 中世 | Day 90 | 産業革命・煙突街(赤レンガ風+煙) |
| modern | 近代 | Day 180 | レンガ造り(茶色矩形) |
| contemporary | 現代 | Day 270 | ガラスビル(青灰色矩形) |
| future | 未来 | Day 365 | 空飛ぶ車・UFO(カラフル) |

#### 進化ロジック
- ユーザーの `started_at` から経過日数を計算
- マイルストーン到達日を超えていれば、新しい時代の建物が混ざってくる
- `milestone_progress.current_era` を更新(サーバーアクションで)

#### 進化の見え方
- 既存の建物は残る(過去の時代のものも見える街にする)
- 新しい時代の建物が**新しく建つときから**新時代の建物として出現
- マイルストーン到達の瞬間は派手な演出(後述)

---

### 3. 季節切り替え

#### 季節の定義

| season | 視覚的特徴 |
|---|---|
| spring | 桜の花びら(ピンクの小円が舞う) |
| summer | 太陽が回転(黄色い円、簡単な回転アニメ) |
| autumn | 焼き芋の煙(灰色の煙が上る) |
| winter | 雪が降る + 雪だるま(白い円が落下) |

#### 切り替えUI
- 街並みの下部に4つのボタン(春/夏/秋/冬)
- クリックで季節が切り替わる
- 季節の状態は**ユーザー設定として保存**(`user_settings.current_season`)し、次回ログイン時も復元
- → これによりDB側にも `current_season` カラムの追加が必要

### 4. 住人NPCのアニメーション(フル実装)

#### 歩行
- 各住人は街の地面ライン上を**横方向に歩く**
- 歩行速度はランダム(1〜2px/フレーム程度)
- 画面端に達したら反転して逆方向に歩く

#### 建物への出入り
- 一定確率(例: 5秒ごとに30%)で建物に入る
- 建物に入っている間は表示されない
- 数秒後に建物から出てきて再び歩く

#### 荷物の増減
- 建物から出てきた時、**ランダムで荷物を持っている**(背中の四角形などプレースホルダー)
- 荷物を持っている時は歩行速度が少し遅くなる
- 別の建物に入ると荷物が消える(配達してきた風)

#### パフォーマンスの注意
- 住人が10人以上いる場合、`requestAnimationFrame` を使い1ループで管理
- Framer Motion の `animate` プロパティで個別管理してもいいが、要素数次第で重くなる
- 適切にメモ化し、不要な再描画を避ける

---

### 5. 建物増加アニメーション(ボイーン)

チェックボックスを完了した瞬間、新しい建物が建つ:

```
scale: 0 → 1.5 → 0.7 → 1.15 → 0.95 → 1
+ キラキラエフェクト(短時間、控えめ)
+ 持続時間: 約0.8秒
```

Framer Motion の `keyframes` で実装。
キラキラエフェクトは小さな星型(div + Tailwind)を建物の周辺にランダム配置 → フェードアウト。

---

### 6. マイルストーン演出(フル実装)

#### Day 1: 建物1棟目が建つ
- 上記「ボイーン」の派手版(scale倍率を大きく、持続時間を長く)
- 画面全体にフラッシュ(白いオーバーレイ、0.3秒)
- メンターから「ようこそ。今日から一緒に街を育てていきましょう」

#### Day 7: 住人1人目登場
- 中央に住人が「ぽん」と現れる(scale 0→1.3→1)
- 建物の隣に立つ
- メンターから「1週間、よく続けられましたね。ささやかですが、住人がひとり訪れました」

#### Day 30/90/180/270/365: 時代進化
- 画面全体にフラッシュ(2秒、派手)
- 既存の街並みが少し暗くなり、新時代の建物が「ジャン!」と現れる
- BGMやSE はオプション(あれば実装、なくてもOK)
- メンターから対応する時代のコメント(後述のメッセージ表参照)

#### Day 365(卒業)
- 通常の時代進化演出 + 追加で「花火」エフェクト(数秒)
- 「卒業」を明示するモーダル or バナー
- 「ここから先は、日数以外の変化はない設計です」と明示する短いメッセージ
- アプリの利用は続けられる旨も伝える

### 7. メンターコメント機能

#### 表示方式
- 街並みの下に常駐するメンター枠
- **顔アイコン(プレースホルダー: 丸+「M」)** + 吹き出し or セリフ枠
- メッセージは5秒で自動消滅 or タップで消せる
- 同時に複数のメッセージは出さない → キューで順次表示
- メッセージがない時は枠だけ表示(枠が消えると寂しいので)

#### コメント定義ファイル(重要)

**コメントを設定ファイルに分離し、ユーザーが後から手動編集・追加できる設計にしてください**。

`lib/mentor/messages.ts`(または `.json`)に以下のような形で定義:

```typescript
// 例: TypeScript の場合
export type MentorMessage = {
  id: string;           // 一意のID
  trigger: string;      // 発火条件のキー
  weight?: number;      // 同じtriggerに複数ある場合の重み(ランダム選択用)
  text: string;         // 表示テキスト
};

export const mentorMessages: MentorMessage[] = [
  {
    id: 'welcome-first',
    trigger: 'first_login',
    text: 'ようこそ。今日から一緒に街を育てていきましょう',
  },
  {
    id: 'daily-return-a',
    trigger: 'daily_return',
    weight: 1,
    text: 'おかえりなさい。今日も少しだけ進めましょうか',
  },
  // ...以下、全コメント定義
];
```

#### 初期コメント一覧(必ず実装するもの)

| trigger | text |
|---|---|
| `first_login` | ようこそ。今日から一緒に街を育てていきましょう |
| `daily_return` | おかえりなさい。今日も少しだけ進めましょうか |
| `check_one_done` | いいですね、一歩進みましたよ |
| `check_all_done` | 今日のノルマ達成です。お疲れ様でした |
| `milestone_day7` | 1週間、よく続けられましたね。ささやかですが、住人がひとり訪れました |
| `milestone_day30` | 30日です。街が時代を進めましたよ |
| `milestone_day90` | 3ヶ月。ここまで来る人はそう多くないですよ |
| `milestone_day180` | 半年。立派なものです |
| `milestone_day270` | 9ヶ月目。あと少しで卒業ですね |
| `milestone_day365` | 1年、お疲れ様でした。ここから先は、あなたのペースで |
| `weekly_achieved` | 今週もきっちり進められましたね |
| `monthly_done` | 今月の記録、確かに受け取りました |
| `monthly_overdue` | 先月の成果、少しだけ振り返ってみませんか |
| `long_absence` | お帰りなさい。街はちゃんとここで待っていましたよ(7日以上ぶり) |

#### トーンの統一原則(コードコメントで明記してください)
- ですます調、丁寧
- 過度な励ましは禁止(「すごい!」「がんばって!」は使わない)
- 機械的にならないよう、語尾に「〜ね」「〜よ」を時々混ぜる
- 数字が動かない時期の人にも刺さらない優しさ
- ビルの受付の女性が、忙しすぎず暇でもなく、ちょうどよく目を合わせてくれる距離感

#### 拡張性
- 同じ`trigger`に複数メッセージを定義できる(`weight` でランダム選択)
- ユーザーが手動でこのファイルを編集・行追加するだけでコメントを増やせる
- メッセージ表示エンジンは、triggerが発火したらこのファイルから該当メッセージを探し、weight付きランダムで選ぶ実装にする

### 8. デバッグツール(フェーズ完了時に削除)

#### 目的
マイルストーン演出やDay数別の見え方を開発中に確認するため。

#### 実装方針(重要)

**フェーズ完了時に削除する前提の設計**にしてください:

- 専用ページ `app/(debug)/day-override/page.tsx` を作る
- このフォルダ全体を**`.env`の`NEXT_PUBLIC_DEBUG_MODE=true` の時だけ有効**にする
- 削除作業は**フォルダ1つを削除すれば完了**するようにファイル境界を切る
- デバッグツールは**サーバーサイドDBを書き換えない**こと
  - クライアント側の状態(Zustand or local state)だけで「Dayを偽装」する
  - DBの`started_at`は変えない
- 本番では絶対に動かない構造にする(`process.env.NODE_ENV === 'development'` でガード)

#### 機能
- Day数を入力 → その日数として街並みが表示される(クライアント側のみで偽装)
- マイルストーン演出を強制発火するボタン(各Day毎)
- メンターコメントを強制発火するボタン(triggerを選択)
- 季節を強制変更するボタン

#### フェーズ完了時の削除手順を README に追記
削除手順を `README.md` の末尾に明記してください:
```
## デバッグツールの削除(フェーズ3完了時)
1. `src/app/(debug)/` フォルダを削除
2. `.env.local` から `NEXT_PUBLIC_DEBUG_MODE` を削除
3. ナビゲーションメニューから debug リンクを削除
```
