## 認証・個人情報設計(確定)

### 方針: 個人情報を一切持たない

メアド・電話番号などPIIをゼロにすることでセキュリティ・プライバシーリスクを最小化する。漏洩しても住所・氏名・連絡先が一切ないので被害が拡大しない。

### 認証方式: ユーザーネーム + パスワード(ダミーメール変換方式)

- ユーザーには「ユーザーネームとパスワード」で見せる
- 内部的には `{username}@dummy.yourapp.com` に変換してSupabase Authに渡す
- メアドは取得せず、サーバー側に個人情報を持たない
- パスワード自体はSupabaseがハッシュ化して保存(自前で持たない)

### パスワード忘れ救済: リカバリーコード方式

- サインアップ時に12桁程度のリカバリーコードを発行
- 「自分で保管してください、なくしたら復旧できません」と明示
- ユーザーがコードでパスワードリセットできる
- サーバー側はハッシュ化したコードのみ保持(原本は持たない)
- 思想は GitHub の 2FA バックアップコードと同じ(責任を本人に渡す)

### 実装時の注意点
- ユーザーネームの重複チェックが必要
- パスワード強度バリデーション必須
- リカバリーコードはサインアップ画面でしか表示しない(再表示なし、ダウンロード/コピー機能を用意)
- ダミードメインは自前のドメインを使うのが望ましい

## Supabase セキュリティ運用ルール(確定)

実装時に厳守するセキュリティルール。Supabase + Next.js 構成で、バイブコーディングのスピードを保ちつつ「Zenn/Qiita/Note」のベストプラクティスに基づいた防御を適用する。

---

### 1. Row Level Security (RLS) ─ 鉄壁の防護

- **Default Deny**: 新しいテーブルを作成したら、まず `ALTER TABLE ... ENABLE ROW LEVEL SECURITY;` を実行し、デフォルトですべてのアクセスを拒否する
- **Public権限の剥奪**: `REVOKE ALL ON public.table_name FROM public;` を検討し、意図しないロールからのアクセスを遮断する
- **Policyの厳格化**: `auth.uid()` を使用し、「自分のデータのみ」にアクセスできるポリシーを確実に設定する
- **anonキーの制限**: ログイン不要のページを除き、`anon` ロール(未認証ユーザー)には一切の権限を与えない

---

### 2. Key Management & Roles

- **Service Role Key**: 絶対にクライアントサイドのコードや環境変数(`NEXT_PUBLIC_*`)に混入させない。RLSをバイパスする「神権限」であることを自覚する
- **Authenticated Role**: ログインユーザーのアクションには必ず `authenticated` ロールを割り当て、ポリシーで制御する

---

### 3. Development Workflow (Claude Code Specific)

- **Security Audit First**: コードを提案する前に、その実装が「RLSを突破されないか」「他のユーザーのデータが漏洩しないか」を内部で監査し、結果を短く報告する
- **SQL Migration**: DB変更時は、Supabaseのダッシュボードで実行可能な「クリーンで再利用可能なSQL」を生成する
- **Performance**: RLSの条件式(USING句)が複雑になりすぎないよう、パフォーマンスに配慮したインデックスの提案もセットで行う

---

### 4. 実装時の指示

- 実装を始める前に、現在のテーブルのRLS状況を確認するために `\d table_name` 相当の確認や、既存ポリシーのチェックを自発的に行う
- ユーザーの利便性とセキュリティがコンフリクトする場合(例: 公開プロフィールの実装など)、必ずリスクを提示した上で、最小権限(Least Privilege)の原則に則った案を出す

## 技術スタック(確定)

| カテゴリ | 採用技術 |
|---|---|
| フレームワーク | Next.js(App Router、TypeScript) |
| バックエンド | Supabase(Auth + Postgres + RLS) |
| パッケージマネージャ | pnpm 11+(Node.js 22+ 必須) |
| スタイリング | Tailwind CSS |
| 状態管理 | Zustand |
| アニメーション | Framer Motion |
| デプロイ | Cloudflare Workers + OpenNext アダプタ(`@opennextjs/cloudflare`) |

### サプライチェーン攻撃対策(確定)

`pnpm-workspace.yaml` で以下を設定:

```yaml
pnpm:
  minimumReleaseAge: 7200       # 5日 = 7200分。これより新しいパッケージはインストールしない
  minimumReleaseAgeStrict: true # 古いバージョンへフォールバックせず、不適合なら失敗
  blockExoticSubdeps: true      # 非標準ソースからの依存解決をブロック(pnpm 11デフォルト)
```

### デプロイ時の注意点

- `@cloudflare/next-on-pages` は **使用しない**(廃止済み)
- 必ず `@opennextjs/cloudflare` を使う
- Next.jsアプリの`edge runtime`指定は外す(OpenNextアダプタは現時点でNode.js runtimeのみ対応)
- `.gitignore` に `.open-next/` を追加
